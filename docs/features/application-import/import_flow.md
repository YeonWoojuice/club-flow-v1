# 지원자 데이터 가져오기 흐름

## 목적

운영진은 선택한 학기에 지원자 데이터를 한 번에 등록할 수 있다. 외부 데이터의 한 행은
CrewCat의 `Application` 한 건으로 변환되며, 이미 같은 학기에 지원한 사람을 중복으로 만들지 않는다.

## 지원하는 입력

현재 지원자 가져오기 화면에서 지원하는 입력은 다음과 같다.

- Google Form 응답이 연결된 Google Sheet
- 일반 Google Sheet

표 파일(CSV·엑셀)을 이용한 지원자 가져오기는 후속 범위다. 잔류 부원 이월용 표 파일 읽기 기능은
구현되어 있지만, 지원자 가져오기 화면에는 아직 연결하지 않는다.

Google Form 자체는 질문을 작성하는 도구이고, 행과 열 형태의 응답 데이터는 연결된 Google Sheet에서 읽는다.
PDF, 이미지, 일반 Google Docs 문서는 표 구조가 일정하지 않으므로 지원하지 않는다.

## 권한과 대상 학기

- `club_staffs.status=APPROVED`인 운영진만 데이터를 가져올 수 있다.
- 운영진은 데이터를 읽기 전에 대상 `Generation`을 반드시 하나 선택한다.
- 선택한 학기에 속한 지원 기록만 생성한다.
- 종료된 학기는 새로운 지원자를 가져올 수 없다.

## 입력 열 연결

파일 또는 Sheet의 첫 행은 열 제목으로 사용한다. 운영진은 미리보기 화면에서 원본 열을
CrewCat 필드와 연결한다.

| CrewCat 필드 | 필수 | 원본 열 예시 |
|---|---:|---|
| 이름 | 예 | 이름, 성명 |
| 이메일 | 예 | 이메일, 이메일 주소, 학교 이메일 |
| 전화번호 | 아니오 | 전화번호, 연락처 |
| 학번 | 예 | 학번 |
| 지원 동기 | 아니오 | 지원 동기 |
| 기타 답변 | 아니오 | Google Form의 나머지 질문 |

- 이름과 이메일 열은 반드시 연결한다.
- 전화번호가 없는 Sheet도 가져올 수 있다. 새 인물을 만들 때 학번은 반드시 필요하다.
- 지원 동기와 기타 질문은 `ApplicationAnswer`로 저장한다.
- 이름·이메일·학번·전화번호·응답 시간으로 연결하지 않은 나머지 열은 질문과 답변으로 자동 저장한다.
- 추가 질문 열이 없는 Sheet도 가져올 수 있다.
- 운영진은 Sheet 문서·고정 탭 ID와 열 연결 설정을 동아리의 저장된 가져오기 설정으로 남길 수 있다.
- 저장 이후 헤더 구성이 바뀌면 잘못된 열이 연결되지 않도록 재설정을 요구한다.

## 데이터 정리와 검증

각 행을 읽은 뒤 저장 전에 다음 규칙을 적용한다.

| 항목 | 규칙 |
|---|---|
| 이름 | 앞뒤 공백을 제거한다. 비어 있으면 오류다. |
| 이메일 | 앞뒤 공백을 제거하고 소문자로 변환한다. 비어 있거나 이메일 형식이 아니면 오류다. |
| 전화번호 | 값이 있으면 원본 값을 저장한다. 없어도 오류가 아니다. |
| 학번 | 앞뒤 공백을 제거한다. 비어 있으면 오류다. |
| 응답 시간 | 값이 있고 시간 형식으로 읽을 수 있으면 원본 시간을 사용한다. 없거나 읽을 수 없으면 확정 시각을 사용한다. |
| 답변 | 추가 질문이 없거나 답변 값이 비어 있어도 저장 실패 사유가 아니다. |

## 중복 규칙

중복 판정에는 정리된 소문자 이메일을 사용한다.

- 같은 원본 안에서 같은 이메일이 두 번 이상 나오면 어느 행이 맞는지 임의로 고르지 않고 해당 이메일 행을 모두 제외한다.
- 같은 동아리의 같은 학기에 이미 같은 이메일의 `Application`이 있으면 새 지원 기록을 만들지 않는다.
- 같은 사람이 다른 학기에 지원하는 것은 허용한다.
- 같은 동아리에 이미 등록된 `Person`이 있으면 그 인물을 재사용한다.
- 기존 `Person`을 재사용할 때 Sheet의 이름·학번·연락처로 기존 프로필을 덮어쓰지 않는다.

## 사용자 흐름

```text
대상 학기 선택
  → 저장된 Google Sheet의 최신 응답 확인 또는 새 주소 입력
  → 원본 열과 CrewCat 필드 연결
  → 행별 검증 및 중복 검사
  → 미리보기와 오류 사유 표시
  → 운영진이 가져오기 확정
  → 정상 행만 Application으로 저장
  → 성공·오류·중복 건수 표시
```

저장 전에는 반드시 미리보기를 보여준다. 운영진이 확정하기 전에는 DB를 변경하지 않는다.

## 결과 표시

가져오기가 끝나면 다음 건수를 구분해 보여준다.

- 저장됨: 새 `Application`이 생성된 행
- 오류: 이름·이메일 누락, 잘못된 이메일처럼 형식이 맞지 않는 행
- 중복: 같은 파일 또는 같은 학기에 이미 존재해 저장하지 않은 행

오류와 중복 행은 원본 행 번호와 사유를 함께 표시한다. 정상 행만 저장하며, 오류 행 때문에
정상 행까지 취소하지 않는다.

## 원본 데이터 보관

MVP에서는 업로드한 파일과 Google Sheet 원본을 별도로 보관하지 않는다. 읽은 값 중
지원자와 답변으로 필요한 데이터만 저장한다.

원본 파일 보관, 가져오기 실행 이력, Google Sheet 주기 동기화는 후속 범위다.

## 현재 구현

1. Google Sheet의 탭·헤더·행을 읽는다.
2. 열 연결 후 오류·원본 중복·같은 학기 기존 지원자를 미리보기로 표시한다.
3. 확정 요청에서 서버가 같은 규칙을 다시 검사한다.
4. 정상 행만 `GOOGLE_FORM` 출처의 `Application`과 선택적인 `ApplicationAnswer`로 저장한다.
5. 자주 쓰는 Sheet는 탭의 고정 ID와 열 연결 규칙을 저장해 다음부터 한 번에 최신 응답을 읽는다.

저장된 설정은 기존 지원자를 자동 수정·삭제하는 동기화가 아니다. Google Sheet 주기 동기화,
표 파일 지원자 가져오기, 가져오기 실행 이력 저장은 후속 범위다.
