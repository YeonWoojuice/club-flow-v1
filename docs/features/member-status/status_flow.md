# 부원 상태 변경 흐름

## 목적

승인된 운영진이 학기별 부원의 현재 활동 상태를 바꾸고, 누가 언제 왜 변경했는지 확인한다.

## 상태 규칙

```text
REGULAR   ↔ ASSOCIATE
REGULAR   ↔ INACTIVE
ASSOCIATE ↔ INACTIVE
INACTIVE  → WITHDRAWN
WITHDRAWN → 변경 불가
```

- `REGULAR`은 모든 활동에 참여하는 회원을 뜻한다.
- `ASSOCIATE`는 일부 활동에 참여하거나 멘토링 등으로 기여하는 준회원을 뜻한다.
- `INACTIVE`는 전반적인 활동을 하지 않는 비활동, `WITHDRAWN`은 탈퇴를 뜻한다.
- 회원과 준회원은 바로 탈퇴할 수 없고 먼저 비활동으로 변경해야 한다.
- 탈퇴는 마지막 상태이며 다시 회원·준회원·비활동으로 바꿀 수 없다.
- 같은 상태를 다시 요청하면 성공으로 처리하지만 변경 이력은 만들지 않는다.
- 탈퇴 사유는 필수이며, 다른 상태 변경 사유는 선택이다.
- 사유는 앞뒤 공백을 제거하고 최대 500자까지 저장한다.

## 처리 흐름

```text
운영진이 부원 행을 눌러 상세 창 열기
  → 기본 정보, 상태 변경, 변경 이력 중 필요한 화면 선택
  → 상태 변경 화면에서 변경할 상태와 사유 입력
  → 회원·준회원의 탈퇴 요청은 비활동을 먼저 거치도록 차단
  → 탈퇴라면 되돌릴 수 없음을 확인
  → 서버가 동아리 접근 권한 확인
  → 서버가 해당 부원 행을 잠금
  → 도메인 상태 규칙 검사
  → 현재 상태 변경 + 변경 이력을 한 트랜잭션으로 저장
  → 화면의 상태 갱신
```

부원 행을 잠그는 이유는 두 운영진이 거의 동시에 상태를 바꿀 때 변경 순서와 이력이 어긋나는 것을
막기 위해서다. 상태 변경과 이력 저장 중 하나라도 실패하면 둘 다 저장하지 않는다.

## 변경 이력

이력에는 다음 내용을 저장한다.

- 이전 상태와 새 상태
- 변경 사유
- 변경한 운영진
- 변경 시간

운영진은 부원 상세 창의 변경 이력 화면에서 최신순으로 조회할 수 있다.

## 권한과 오류

- `club_staffs.status=APPROVED`인 운영진만 해당 동아리 부원의 상태를 변경하거나 이력을 조회할 수 있다.
- 다른 동아리 부원에는 접근할 수 없다.
- 탈퇴 상태를 되돌리려 하면 충돌 오류로 처리한다.
- 잘못된 상태 문자열이나 깨진 요청은 공통 오류 형식 `{ "code": "...", "message": "..." }`으로 반환한다.
