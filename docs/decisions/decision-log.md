## 2026-07-08 ~ 07-09 · 협업 구조 개편

### 1. 분업 기준: 도구별 레이어 → 기능(수직) 단위

- **결정**: "UI는 Claude, 백엔드는 Codex" 방식을 버리고, 한 기능을 한 도구가 프론트+백 끝까지 맡는다.
- **이유**: 경계선이 코드 한가운데(API 계약)를 지나가면 양쪽이 각자의 방언으로 진화한다.
  실제로 에러 처리 분열, `Input` vs `Request` 네이밍, phone 필수 여부 불일치가 발견됐다.
- **버린 대안**: 레이어 분업 유지 + 자동화 보완 — 타입은 지켜주지만 스타일·철학 분열은 못 막는다.

### 2. AI 역할: Claude = 설계·리뷰·막힌 문제, Codex = 구현

- **결정**: Claude가 설계를 문서로 남기면 Codex가 그 문서와 AGENTS.md만 읽고 구현한다.
  계약 변경·인증·마이그레이션은 완료 후 Claude 리뷰를 거친다.
- **이유**: 1. Claude 토큰 소모가 빠르다 → 비싼 도구는 판단이 필요한 일에만.
  2. 두 도구를 대화로 중계시키는 노동이 컸다 → 전달을 대화가 아닌 문서로 바꿔 중계 비용 제거.
- **버린 대안**:
  - 기능별 전담 + 교차 리뷰 — 핸드오프마다 사용자가 중계해야 해서 기각.
  - 단일 도구 통일 — 두 번째 시각을 잃고, 비용 분산도 안 됨.
  - Claude 단독 — Cowork 샌드박스의 실행 한계(테스트·빌드를 사용자에게 떠넘김)와 토큰 비용.

### 3. 규칙 문서: AGENTS.md 단일 소스

- **결정**: AGENTS.md가 모든 규칙·구조·용어의 원본. CLAUDE.md는 참조 스텁으로 축소.

- **이유**: 어떤 도구가 와도 같은 문서를 읽으면 규칙이 도구별로 갈라질 통로가 닫힌다.
  실제로 스택 표가 4개 문서에 중복돼 "Spring Boot 4" vs "4.0.7"로 이미 어긋나 있었다.

- **버린 대안**: 수동 타입 유지 + 조심 — phone 불일치가 이미 그 방식의 실패 사례.

### 5. 에러 문구의 단일 소스 = 백엔드

- **이유**: 에러 문구를 양쪽에서 관리하면 중복. 백엔드는 이미 잘 만들고 있었는데 프론트 조회 경로가 버리고 있었다.

## 2026-07-11 · 인증, 외부 데이터 가져오기, 중복 방지

### 11. CrewCat 로그인과 Google Sheet 읽기 권한을 분리

- **결정**: CrewCat 로그인은 서버 세션(`JSESSIONID`)으로 처리하고, Google Sheet 읽기는
  `spreadsheets.readonly` 범위를 별도로 동의받는다. Google 접근·갱신 토큰은 백엔드 DB에
  AES-256-GCM으로 암호화해 저장하며 프론트엔드에는 전달하지 않는다.
- **이유**:
  1. 로그인만 필요한 사용자에게 불필요한 Sheet 권한 동의를 요구하지 않는다.
  2. 중요한 Google 토큰이 브라우저 저장소나 JavaScript에 노출되는 것을 막는다.
  3. CrewCat 로그인 상태를 유지하면서 Sheet 연결만 별도로 해제할 수 있다.
- **버린 대안**:
  - 최초 Google 로그인에서 Sheet 권한까지 한꺼번에 요청 — 로그인 진입 장벽과 권한 범위가 커진다.
  - Google 접근 토큰을 브라우저에 저장 — 탈취 시 Google 데이터에 직접 접근할 위험이 있다.

### 12. CSRF 토큰은 메모리에 캐시하고, 쓰기 요청의 첫 403에서 한 번만 갱신

- **결정**: 프론트엔드는 CSRF 토큰을 실행 중인 메모리에만 캐시한다. 로그아웃이 끝나면 성공 여부와
  관계없이 캐시를 비우며, 쓰기 요청이 403이면 토큰을 폐기하고 새 토큰으로 원래 요청을 정확히 한 번
  재시도한다. 재시도도 403이면 실제 권한 오류로 처리한다.
- **이유**:
  1. 쓰기 요청마다 토큰을 다시 발급받는 불필요한 통신을 줄인다.
  2. 로그아웃·재로그인이나 세션 변경 후 이전 토큰이 남은 상황을 자동으로 복구한다.
  3. 한 번만 재시도해 무한 반복과 권한 없는 요청의 반복 전송을 막는다.
- **버린 대안**:
  - 쓰기 요청마다 항상 새 CSRF 토큰 발급 — 안전하지만 불필요한 요청이 계속 발생한다.
  - 403에서 무제한 자동 재시도 — CSRF가 아닌 권한 오류에서도 반복 요청이 발생한다.
  - 403을 즉시 최종 실패 처리 — 세션 변경 후 사용자가 새로고침하기 전까지 정상 요청도 실패한다.

### 13. Google Sheet 가져오기는 자동 동기화가 아닌 일회성 스냅샷으로 처리

- **결정**: Google Form 응답 Sheet와 일반 Google Sheet를 같은 표 데이터로 읽고, 사용자가 사용할 탭과
  열을 선택한 뒤 미리보기에서 정상으로 판정된 행만 DB에 저장한다. 가져온 뒤 원본 Sheet와 자동으로
  동기화하지 않으며, MVP에서는 원본 행·파일·가져오기 이력·열 연결 설정을 별도로 보관하지 않는다.
- **이유**:
  1. 동아리와 조사 목적마다 열 구성이 달라 프로그램이 정답 열을 안전하게 추측할 수 없다.
  2. 원본 수정이 CrewCat의 확정 데이터까지 자동 변경하면 데이터 책임과 오류 복구가 복잡해진다.
  3. 자동 동기화·이력 관리·매핑 재사용을 함께 만들면 MVP 범위가 크게 늘어난다.
- **버린 대안**:
  - 고정된 하나의 Sheet 양식을 강제 — 기존 동아리 문서를 활용하기 어렵다.
  - Google Form 문서 자체를 분석 — 실제 응답 행렬은 연결된 Sheet에 있으므로 실익이 없다.
  - Sheet를 주기적으로 자동 동기화 — 충돌 해결과 삭제·수정 정책이 먼저 필요하다.
- **현재 한계**: 주기 동기화, 원본 보관, 가져오기 이력, 열 연결 설정 재사용은 후속 범위다.

### 14. 사람은 동아리 안에서 정규화된 이메일로 식별

- **결정**: `Person`은 `(club_id, email)`로 고유하게 식별한다. 이메일은 공백을 제거하고 소문자로
  정규화한다. 같은 사람의 학기별 지원과 부원 기록은 각각 `(generation_id, person_id)`로 중복을 막는다.
- **이유**:
  1. 같은 사람이 여러 학기에 참여해도 하나의 `Person`에 이력을 연결할 수 있다.
  2. 이름은 중복될 수 있고 연락처·학번은 변경되거나 누락될 수 있다.
  3. Google Form과 운영진 입력에서 이메일을 공통 비교 기준으로 사용하기 쉽다.
- **버린 대안**:
  - 학번만으로 식별 — 학교·입력 형식 차이와 누락을 처리하기 어렵다.
  - 이름과 연락처 조합으로 식별 — 동명이인과 연락처 변경에 취약하다.
  - 학기마다 새 `Person` 생성 — 같은 사람의 학기별 이력이 분리된다.
- **현재 한계**: 사용자가 이메일을 변경했을 때 기존 `Person`과 자동으로 병합하지 않는다. 이메일 변경과
  인물 병합 정책은 후속 결정이 필요하다.

### 15. 중복과 동시 요청은 서버·DB에서 최종 보장

- **결정**: 프론트의 버튼 비활성화는 사용자 편의용으로만 사용한다. Service에서 현재 상태를 다시
  확인하고, DB 고유 제약을 마지막 안전장치로 둔다. 같은 데이터를 동시에 변경할 위험이 있는 지원 상태,
  부원 이월·상태, 운영진 초대 수락·역할·접근 해제에는 필요한 행만 비관적 잠금으로 직렬화한다.
- **이유**:
  1. 화면에서 중복 클릭을 막아도 API 직접 호출과 여러 운영진의 동시 요청은 막을 수 없다.
  2. 조회 후 저장 사이에 다른 요청이 끼어들 수 있으므로 애플리케이션 검사만으로는 부족하다.
  3. DB 제약과 행 잠금을 함께 사용해야 최종 데이터의 중복과 상태 불일치를 막을 수 있다.
- **버린 대안**:
  - 프론트 버튼 비활성화만 사용 — 다른 브라우저·사용자·직접 API 호출을 막지 못한다.
  - Service의 사전 조회만 사용 — 동시 요청이 같은 기존 상태를 읽고 모두 저장할 수 있다.
  - 모든 조회에 잠금 사용 — 불필요한 대기와 성능 저하가 생기므로 변경 경쟁이 있는 경로에만 적용한다.

### 16. 운영진은 가입 전에 Google 이메일로 초대 가능

- **결정**: 아직 CrewCat에 가입하지 않은 사람도 Google 이메일로 부회장·운영진 초대를 받을 수 있다.
  수락·거절 시 로그인으로 확인된 이메일과 초대 이메일을 비교한다. 같은 동아리·이메일의 대기 초대는
  하나만 허용하고, 접근이 해제된 사용자가 재초대를 수락하면 기존 `ClubStaff` 행을 다시 승인한다.
- **이유**:
  1. 초대 전에 상대방에게 CrewCat 가입을 요구하지 않아도 된다.
  2. Google 로그인으로 이메일 소유자를 확인한 뒤에만 권한을 부여할 수 있다.
  3. 기존 권한 행을 재사용하면 같은 사용자·동아리의 권한 데이터가 여러 행으로 갈라지지 않는다.
- **버린 대안**:
  - 가입한 사용자만 검색해 초대 — 초대 전에 별도 가입 절차를 요구한다.
  - 비밀 초대 링크로 수락 — 링크 토큰의 발급·만료·전달 보안과 메일 발송 인프라가 추가로 필요하다.
  - 재초대 때 새 `ClubStaff` 행 생성 — `(club_id, user_id)` 고유성과 권한 이력의 기준이 복잡해진다.
- **현재 한계**: 자동 초대 메일 발송은 현재 범위에 포함하지 않는다. 초대받은 사용자는 동일한 Google
  이메일로 로그인한 뒤 받은 초대 화면에서 확인한다.
